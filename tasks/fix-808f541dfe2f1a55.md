# fix-808f541dfe2f1a55 — E1T4 bypasses toolkit storage with direct Redis, completely isolated from all other E1 handlers

**Weight:** 0.0000 (share of project budget)

`E1T4.ts` loads `ioredis` directly via `createRequire` and connects to Redis through its own code — it does **not** use the toolkit's `resolveSessionStorage` or `counter-store.ts`. Key scheme is `tallybot:{userId}:counters` as an HSET (hash fields). This means `/reset` cannot reach any counter created by `/new`, `/inc`, or readable by `/get` or `/list` — it's a silo. Even if all handlers pointed to the same Redis, the key namespace mismatch (`tallybot:{userId}:counters` fields vs `counter:<name>` top-level keys) would prevent data sharing. **Additionally**, the handler lacks a per-user namespace in its Redis keys — other handlers use global `counter:<name>`, so even conceptually `/reset` should not be user-scoped unless all handlers are.
