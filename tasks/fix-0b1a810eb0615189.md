# fix-0b1a810eb0615189 — E1T2 uses isolated session storage with incompatible schema, breaking cross-command data flow

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 TALLY

`E1T2.ts` calls `resolveSessionStorage<Record<string,number>>(undefined)` in module scope, creating a **new, separate** `MemorySessionStorage` (or `RedisSessionStorage`) instance. It stores **all** counters as a single JSON record under key `"inc:counters"` (e.g. `{visitors:1, bugs:10}`). This is fundamentally incompatible with:
- `counter-store.ts` (E1T1 `/new`, E1T5 `/list`) which stores one counter per key as `counter:<name> → {name,value}`
- `stores/counter.ts` (E1T3 `/get`) which uses the same `counter:<name>` prefix but a separate storage instance

**Result:** `/new test1` creates a counter in one store; `/inc test1` looks in a different one (finds nothing, silently initializes to 0+1). The entire create→increment→get→list workflow is broken. This is **not** caught by E1T2's test spec because the tests only exercise `/inc` in isolation (no cross-command scenario).
